# OSCA 开放规范白皮书写作设计

> 日期：2026-07-12
> 状态：已完成方向讨论，待书面设计复核
> 目标产物（提议）：`docs/OSCA-WHITEPAPER-v0.1.zh-CN.md`

## 1. 目标

编写一份以 **OSCA 开放规范**为主角、以 **Oscaware 作为参考实现**的中文白皮书。

第一读者是关注 AI Agent 的产品与技术同行。白皮书的首要转化目标不是介绍
Oscaware 产品，而是让读者理解 OSCA 的问题意识、设计取舍和运行逻辑，并愿意：

1. 阅读 OSCA SPEC；
2. 运行公开样例；
3. 基于 OSCA 规范创建自己的 Agent；
4. 实现兼容的 Runtime、Connector 或反馈飞轮。

## 2. 开篇第一性假设

白皮书从以下组织观出发：

> **AI Native 组织是一条认知工作流水线——AI 是稳定运转的流水线本身，专家是线上的判断节点，同时是线外的作者与所有者。**

由此建立 OSCA 与主流“Agent 是数字员工”叙事的分歧：

- “数字员工”隐喻容易把 Agent 理解成一个被随机叫起、接收任务、完成应答后结束的主体；
- OSCA 把 Agent 理解成一条有明确目标、稳定结构、类型化连接和确定触发条件的认知工作流水线；
- 专家不是被 AI 替代的旧流程角色，而同时拥有两种身份：
  - **线上的判断节点**：在关键裁决点终审、审批、纠正；
  - **线外的作者与所有者**：专家的真实反馈经确认后成为可署名的贡献账本，持续改写流水线的判断边界；
- 因此 OSCA 关注的不是一次对话是否聪明，而是一条认知工作流水线是否能运行、
  受控、积累裁决并在环境变化后继续贴合现场。

开篇不应把这一定义写成已经得到证明的组织理论。它是 OSCA/Oscaware 的设计假设，
后续需要由真实场景中的返工量、判断复用和专家接受度检验。

## 3. 核心定位

建议正文定义：

> **OSCA 是一种用纯文本定义 AI 认知工作流程的开放规范。它描述工作目标、组合步骤、外部能力与触发时机，并允许经真实反馈确认的人类判断在后续运行中自动生效。**

关系定义：

```text
OSCA
开放规范与运行契约

Oscaware
OSCA 的参考工具、Runtime 与反馈飞轮实现
```

白皮书必须始终保持这个主从关系，不写成 Oscaware 产品功能手册。

## 4. 叙事方案

采用已确认的 **B 为主、C 为辅**：

- 主线采用“问题 → 原理 → 规范 → 运行 → 飞轮 → 参考实现 → 创建自己的 Agent”；
- 用“月度经营诊断 Agent”贯穿全书，展示专家 diff 如何形成 Judgment；
- 用“临期商品定价 Agent”作为侧栏案例，展示 decision vs reality 的 outcome 证据；
- 案例用于解释通用机制，不能让读者误以为 OSCA 只适用于报告生成或经营诊断。

## 5. 证据纪律

全书区分四个证据等级：

| 等级 | 含义 |
|---|---|
| 设计原则 | OSCA 主张应该怎样工作 |
| 已实现 | Oscaware 参考实现已有对应代码和测试 |
| 已演练 | 已使用合成或脱敏样例跑通 |
| 已验证 | 已由真实业务、真实专家与持续数据证明 |

措辞要求：

- 不把“有测试”写成“已证明业务有效”；
- 不把 mock replay 绿灯写成真实模型遵循；
- 不使用“越用越聪明”“自我进化”等无条件承诺；
- P0 完成前，对飞轮价值的措辞上限是“机制与早期演练”；
- 对当前缺口（整本回放、真实 Connector、Host 语义检索接入、P0）明确标注。

## 6. 白皮书与其他文档的边界

| 文档 | 回答的问题 |
|---|---|
| 白皮书 | 为什么需要 OSCA、核心模型是什么、各角色怎样协作、如何开始创建 Agent |
| OSCA SPEC | 文件字段、约束、运行语义和兼容要求的精确定义 |
| Lint 规则 | 哪些纪律由机器检查，以及错误/警告口径 |
| 技术黄皮书 | 参考实现模块、算法、接口、部署、并发与失败恢复 |
| README | 安装、命令和快速运行 |

白皮书可以使用概念目录、简化 YAML、伪代码和架构图，但不复制完整 Schema、API
或逐文件代码说明。

## 7. 章节结构

### 前言：这份白皮书写给谁

说明目标读者、读完后的行动、本文非目标和证据等级。开篇即呈现“认知工作流水线”
假设，给读者一条清楚的阅读主线。

### 第一部分：为什么需要 OSCA

#### 第 1 章：Agent 不是数字员工，而是认知工作流水线

从两种 Agent 隐喻的分歧切入。解释为什么 OSCA 从目标、结构、连接与触发出发，
以及专家为何同时是线上判断节点与线外作者/所有者。提出“现有 Agent 缺少判断层”
这一问题，而不是否定工作流、RAG、工具调用或多 Agent 技术。

#### 第 2 章：OSCA 的核心主张

给出 OSCA 定义，区分 OSCA 与 Oscaware，明确 OSCA 不是模型、Prompt 模板、RAG
框架或自主立法系统。将“进化”改写为“有证据的适应”。

### 第二部分：OSCA 如何定义一个 Agent

#### 第 3 章：O/S/C/A/J 五层定义

解释 Object、Structure、Connector、Aware 与 Judgment。O/S/C/A 定义工作的稳定结构，
J 表示运行中积累的裁决权。使用月度经营诊断案例逐层映射。

#### 第 4 章：`.osca` 包——可交付的 Agent 资产

介绍纯文本包、Git 账本、开发态与交付态、劝告与笼子分离、文件是真相而索引是缓存，
以及客户对 Agent 定义和账本的所有权。

### 第三部分：OSCA Agent 如何运行

#### 第 5 章：双平面运行模型

解释确定性控制平面与短命认知 Episode。按 Trigger → Gate → Judgment 检索 → Episode
装配 → Pipeline → Human/Outcome 展开一次完整运行，介绍五类 Performer 与三级停。

### 第四部分：贡献账本与反馈飞轮

#### 第 6 章：从反馈到判断资产

定义 Case、Candidate、Judgment，解释 Judgment 五段式、只追加、Supersedes、Trust、
正/负判断同权，以及为什么专家确认不等于高 Trust。

#### 第 7 章：两种证据——人类反馈与现实回声

分别解释专家 diff 和 decision vs reality。主案例使用经营诊断 diff，侧栏使用临期定价
outcome，说明“现实是第二位专家”。

#### 第 8 章：归纳给 AI，拍板给人

完整解释 Capture → Distill → Confirm → Ledger → Index → Retrieve → Replay。重点是
Evidence/Meta 由机器管账、AI 只能归纳、专家负责立法、Runtime 执行、回放验证。

### 第五部分：安全、审计与迁移

#### 第 9 章：为什么 Judgment 留在文本层

比较文本账本与权重记忆，解释追溯链、Manifest/Binding/Executor、安全笼子、模型迁移、
整本回放和 Kill switch。

### 第六部分：创建自己的 OSCA Agent

#### 第 10 章：什么场景适合 OSCA

给出重复性、判断密度、反馈闭环三项准入问题，说明首个 Agent 应优先选择高频、
反馈快、错误可控的场景，并列出不适用场景。

#### 第 11 章：从空账本创建第一个 Agent

按 Object → Structure → Connector → Aware → AGENT/Policy → lint/pack/load → Episode
→ capture/distill/confirm → index/replay 给出方法级教程。强调第一版允许空账本，禁止
凭空伪造专家 Judgment。

### 第七部分：Oscaware 参考实现

#### 第 12 章：从规范到可运行系统

说明参考实现的作用、公开与私有边界、规范如何被实现反哺，以及当前能力属于“已实现、
已演练、已验证”的哪一级。状态快照注明日期，详细变化链接到在线文档。

### 第八部分：生态、评估与开放问题

#### 第 13 章：兼容实现与 Agent 组合

讨论格式、Lint、Runtime、Replay 兼容，Agent 作为 Connector、MCP/企业能力层，以及
未来规范治理、认证和第三方扩展问题。

#### 第 14 章：如何判断 OSCA 是否真的有效

解释被确认判断裁决数、返工量曲线、推翻率、覆盖率、Candidate 确认率、Replay 红灯率
和专家额外负担。说明 Phase 0 是业务生死验证，不是组件数量验收。

#### 第 15 章：诚实边界与开放问题

列出蒸馏质量、Guard 表达力、多专家冲突、慢场景供料、Outcome 蒸馏、模型迁移、跨包
复用、规范治理等未决问题。结尾把下一步收束到 M3 工程收官与 P0 真账本。

### 结语：把人的裁决留在系统里

回到克制主张：Agent 的长期价值不仅是第一次生成得多快，而是人的真实裁决能否在
下一次恰当情境中重新生效，并始终可检查、推翻和迁移。邀请读者创建 OSCA Agent、
运行样例、提出规范问题和实现兼容工具。

### 附录

- A：术语表；
- B：最小 `.osca` 包示意；
- C：十条设计公理白皮书短版；
- D：白皮书、SPEC、Lint、黄皮书和 README 导航；
- E：版本与证据声明。

## 8. 编辑规范

1. 每章首先回答一个问题，不从字段清单开始；
2. 每章至少有一个“开发者带走什么”的小结；
3. 抽象机制尽量回到贯穿案例；
4. 主体使用概念图、时序图和小型对照表，避免大段代码；
5. 第一次出现的术语给中文解释与英文名，之后保持一致；
6. 对“数字员工”叙事提出不同判断，但不树立稻草人，不否定适合该模式的场景；
7. 对第三方实现保持中立：Oscaware 是参考答案，不是唯一合法实现；
8. 当前实现状态与永续设计分开，避免白皮书快速过期。

## 9. 写作顺序

正文不按目录机械顺序一次写完，建议按以下批次审阅：

1. 摘要、开篇、第 1～2 章：先把核心命题和边界写准；
2. 第 3～5 章：定义与运行模型；
3. 第 6～9 章：账本、飞轮、安全与迁移；
4. 第 10～12 章：开发者采用路径与参考实现；
5. 第 13～15 章、结语和附录：生态、评估、开放问题；
6. 全文统一术语、证据等级、案例和交叉链接。

每批经用户红笔后再进入下一批，防止开篇立场变化导致整篇重写。

## 10. 完成标准

- 读者能在不阅读代码的情况下复述 OSCA 解决的问题；
- 能清楚区分 OSCA、Oscaware、普通编排、RAG 与模型微调；
- 能解释 O/S/C/A/J、双平面、Case/Candidate/Judgment 和反馈飞轮；
- 能判断自己的场景是否适合 OSCA；
- 能沿第 11 章与 SPEC 创建第一个空账本 Agent；
- 全文没有把合成测试或机制演练写成真实业务证明；
- 所有当前状态声明有日期和来源，所有精确技术约束链接到 SPEC/黄皮书。
